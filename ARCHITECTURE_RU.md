# Архитектура и Логика Пайплайна Дедупликации Новостей

Данный документ подробно описывает техническую логику работы гибридного пайплайна для дедупликации и кластеризации новостных сообщений. Решение спроектировано с учетом требований высочайшей масштабируемости (100 000+ новостей), высокой семантической точности и полной информационной безопасности (100% offline-режим).

---

## 1. Строгая Валидация Данных (Data Ingestion & Pydantic)
На этапе загрузки данных используется библиотека `Pydantic`.
- **Зачем это нужно:** Новостные потоки из Telegram, СМИ и агрегаторов часто содержат "мусорные" данные или пропущенные поля.
- **Логика:** Модель `NewsArticle` строго типизирует входящий JSON. Дата публикации (`published_at`) автоматически парсится в Python-объект `datetime`. Любые невалидные записи безопасно отбрасываются.

## 2. Этап 1: Семантическая Векторизация и Скользящее Окно (Embeddings & FAISS)
Классические подходы плохо справляются с перефразированием.
- **Модель:** `intfloat/multilingual-e5-base`.
- **Логика:** Каждая новость прогоняется через модель Transformer (1.1 ГБ), получая плотный вектор. 
- **Скользящее окно (Sliding Window):** Сравнение 100 000 новостей дает 10 миллиардов пар ($O(N^2)$). Мы используем временное окно **±2 дня**. Нет смысла сравнивать новости с разницей в год. Это радикально снижает сложность.
- **Векторный поиск (FAISS):** Векторы нормализуются (`L2`), и для поиска ближайших соседей применяется порог косинусного сходства **`0.96`**.

## 3. Этап 2: Лексический фильтр (MinHash Veto)
Плотные векторы (E5) могут "галлюцинировать" (давать высокий скор статьям-антагонистам с одинаковым лексиконом, например, "Apple покупает компанию" и "Компания покупает Apple"). 
- **Логика:** Применяется алгоритм локально-чувствительного хеширования (LSH) — MinHash.
- **Действие:** Он оценивает пересечение лексики (Jaccard Similarity). Если Jaccard < 0.1 (статьи вообще не имеют общих слов), пара жестко отсекается (Veto), не допускаясь до тяжелой проверки. Это экономит часы машинного времени.

## 4. Этап 3: Глубокая семантическая проверка (Cross-Encoder)
Отобранные пары-кандидаты проходят финальную проверку.
- **Модель:** `BAAI/bge-reranker-v2-m3`.
- **Логика:** Cross-Encoder читает обе статьи *одновременно*, используя механизм self-attention между токенами обеих статей. Это дает невероятно высокую точность выявления логических противоречий. Если оценка $\ge 0.8$, пара считается истинными дубликатами.

## 5. Этап 4: Графовая Кластеризация (Louvain Community Detection)
Подтвержденные дубликаты образуют граф.
- **Проблема (Topic Drift / Snowballing):** Транзитивное замыкание (A похоже на B, B на C $\rightarrow$ A и C склеиваются) приводит к "снежному кому" тем. В ранних версиях это объединяло тысячи разных финансовых отчетов.
- **Решение:** Алгоритм **Louvain** ищет плотные сообщества (комьюнити) внутри графа. Он математически оценивает плотность связей (Modularity) и разрезает слабые транзитивные мостики, оставляя только те кластеры, внутри которых статьи максимально сильно связаны друг с другом.
- **Результат:** Идеальная изоляция новостных сюжетов.

## 6. Выбор Канонической Новости
Согласно требованиям, для каждого кластера выбирается одна каноническая новость.
- **Логика:** Алгоритм итерируется по всем узлам (ID новостей) внутри сгенерированного кластера и выбирает ту новость, чье поле `published_at` является самым ранним (minimum datetime). Таким образом, первоисточник всегда признается каноническим.

Пайплайн сохраняет результат своей работы в файл `data/clusters_output.json`.
Формат:
```json
[
  {
    "cluster_id": 0,
    "canonical_news_id": "104",
    "news_ids": ["104", "105_synth", "299"]
  }
]
```

## 7. Генерация Эталонного Датасета (Cleaned Dataset)
Утилита `export_cleaned.py` читает файл кластеров и извлекает только канонические статьи, сохраняя их в `news_dump_cleaned.json`. Это позволяет одним скриптом получить кристально чистый массив данных, где от каждого события или даже мусорной заглушки сайта осталась ровно 1 копия.
